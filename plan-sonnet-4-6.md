## Honest Critique First

**Where you're right:**
- The gap is real. Ask Claude "what does Ashtavakra Gita say about the self?" and it blends, paraphrases, and hallucinates citations. A grounded system citing verse+chapter is genuinely differentiated.
- The corpus is publicly available and large enough to build a serious product.

**Where you're wrong or at risk:**

1. **Osho and Sadhguru are off-limits.** Osho International Foundation has registered copyrights in 40+ countries and actively pulls YouTube videos, books, and apps. Isha Foundation has a similar stance. Crawling those transcripts for a commercial AI product is a lawsuit, not a gray area. Cut them from scope.

2. **Fine-tuning is the wrong primary tool for this problem.** Fine-tuning teaches a model *how to behave*, not *what to know*. Research from EMNLP 2024 shows FT on documents barely outperforms the base model on factual recall and actively forgets the long tail — exactly the deep texts you care about (Manu Smriti, Ashtavakra Gita). The moment you fine-tune, you cannot audit which fact came from which source. For a product where the value proposition IS accurate citation, this is architecturally backwards.

3. **100 GPU hours on Colab won't beat a well-built RAG.** You'd spend weeks fine-tuning a 7B model with QLoRA and end up with a system that hallucinates Sanskrit verse numbers. Use those GPU hours later for optional style fine-tuning (how the model *talks*, not what it *knows*).

4. **Don't build the app first.** Validate the retrieval quality and conversation quality before you build iOS/Android. Many spiritual AI projects die because the retrieval is weak and answers feel generic. Prove the core works first.

## Phase 1 — Data Corpus

- **Bhagavad Gita** — Edwin Arnold (1885), Kashinath Telang in Sacred Books of the East (SBE) Vol. 8

- **Manu Smriti** — Georg Bühler, SBE Vol. 25 (1886)

- **Brahma Sutras + Shankara's commentary** — G. Thibaut, SBE
- **Mahabharata** — Kisari Mohan Ganguli (1883–96), only complete PD English translation
- **Ramayana** — Ralph Griffith (1895)
- **Vivekachudamani** — Madhavananda (1921)
- **Vivekananda Complete Works** — Original writings (d. 1902), fully public domain
- **Sanskrit originals** — GRETIL (Göttingen) for all of the above; enables dual-language citation

### Tier B — Use after legal check
- Ramana Maharshi: died 1950, PD in India since ~2011. Pre-1931 publications are safe. Sri Ramanasramam's modern English editions are NOT.
- Wisdomlib.org: Has structured chapter/verse navigation for many PD texts. Some modern works mixed in — verify per book.

### Sources to ingest from
- [sacred-texts.com](https://sacred-texts.com) — SBE full shelf, Ganguli Mahabharata, Griffith Ramayana
- [Project Gutenberg](https://gutenberg.org) — Arnold Gita, Johnston Yoga Sutras, Griffith Ramayana
- [GRETIL](https://gretil.sub.uni-goettingen.de) — Sanskrit originals, verse-level, TEI-normalized
- [Wikisource](https://en.wikisource.org) — Vivekananda Complete Works, SBE volumes
- [Internet Archive](https://archive.org) — SBE, Madhavananda Vivekachudamani, verify `NOT_IN_COPYRIGHT` per item

### Do NOT use
- Osho / OIF: copyrighted in 40+ countries, active enforcement
- Sadhguru / Isha Foundation: copyrighted, personality rights protected by Delhi High Court
- J. Krishnamurti: KFT/KFA sole copyright, d. 1986
- Sri Aurobindo Ashram Trust: claims copyright on Complete Works
- Prabhupada / BBT translations (court-enforced in India)
- Ramamoorthy/Nome's Ribhu Gita (SAT copyright)

## Phase 2 — Architecture (RAG, not fine-tuning)

**Verdict: RAG is correct for this product. Fine-tuning is optional and secondary.**

The reason: your value proposition is "the model correctly cites Manu Smriti chapter X, verse Y" — RAG retrieves the actual passage; fine-tuning memorizes compressed phrasings with frequency bias and forgets the long tail.

```
User query
    │
    ▼
Query understanding layer
(intent + scripture scope detection)
    │
    ├── Hybrid retrieval
    │   ├── BM25 (exact Sanskrit term / verse number match)
    │   └── Dense embeddings (paraphrase / semantic similarity)
    │           └── Vector DB (Qdrant or Weaviate)
    │
    ▼
Re-ranker (cross-encoder, e.g. cohere-rerank or BGE reranker)
    │
    ▼
LLM generation (constrained to retrieved context only)
    │
    ▼
Response with citations: {text, chapter, verse, translator, source}
    │
    ▼
Entailment check (verify claim is grounded in retrieved chunks)
```

### Chunk schema (critical — do this right from the start)
Every chunk must carry:
```json
{
  "text": "...",
  "scripture": "Ashtavakra Gita",
  "chapter": 1,
  "verse": 12,
  "translator": "John Richards",
  "year": 1994,
  "language": "en",
  "license_tier": "A",
  "source_url": "https://..."
}
```
Never mix translators in one retrieval result without labeling. "Gita 2.47" has different phrasings in Arnold vs Telang — the user deserves to know which they're reading.

### Tech stack
- **Embeddings**: `text-embedding-3-small` (OpenAI) or `BAAI/bge-m3` (open, multilingual, handles Sanskrit transliteration)
- **Vector DB**: Qdrant (open source, self-hostable, production-grade)
- **LLM**: Start with Claude Haiku / GPT-4o-mini via API (cheapest, best instruction-following). Optionally switch to self-hosted Llama 3.1 70B later.
- **Orchestration**: LangChain or LlamaIndex (LlamaIndex has better document chunking for structured texts)
- **BM25**: Elasticsearch or Typesense for keyword/exact verse lookup
- **Backend**: FastAPI (Python)
- **Frontend web**: Next.js
- **Mobile**: React Native (shares logic with web)

### Optional fine-tuning (Phase 3 or later)
Use 100 Colab GPU hours for **RAFT** (Retrieval-Augmented Fine-Tuning): teach the model to *read retrieved chunks better* and adopt a reflective, philosophical tone. NOT as a knowledge store. Do this only after you have 500+ real user queries to train on.

## Phase 3 — Product & Conversation Design

The differentiation is not just which texts you have — it's *how* the conversation feels. The model should:
- Not answer like a Google search result
- Speak in a contemplative register (reflective, not clinical)
- Offer multiple perspectives when texts disagree (Advaita vs Dvaita on the same question)
- Always surface the source so the user can read more

Conversation modes to design:
1. **Ask a question** — grounded Q&A with citation
2. **Explore a text** — "walk me through Ashtavakra Gita chapter 2"
3. **Situational guidance** — "I'm anxious about a decision" → relevant shlokas surfaced
4. **Compare traditions** — "how does Advaita differ from Samkhya on consciousness?"

## Phase 4 — Hosting & Deployment

### MVP (validate first, spend last)
- **Backend**: Railway or Render (Python FastAPI) — free tier to start
- **Vector DB**: Qdrant Cloud free tier (1GB) or self-host on a $10/mo VPS
- **LLM**: API calls (no GPU needed for RAG)
- **Web**: Vercel (Next.js, free tier)
- **Cost at 1K users/day**: ~$50–150/mo in LLM API costs at haiku/mini pricing

### Scale (when validated)
- Move vector DB to self-hosted Qdrant on a 4-core VPS (~$20/mo)
- Add Redis for caching frequent verse lookups
- Mobile: React Native deployed to App Store / Play Store
- CDN: Cloudflare in front of everything

### Do NOT do at MVP stage
- Self-hosted LLM (unnecessary complexity, GPU cost)
- Custom fine-tuned model (premature)
- Separate iOS/Android native apps (React Native covers both)
- Your own embedding model (API is fine for now)

## Phased Execution Order

1. **Data pipeline** — ingest, chunk, embed Tier A corpus with correct metadata
2. **Retrieval quality** — tune hybrid search, validate that "Gita 2.47" actually returns the right verse
3. **Generation layer** — constrained prompting with citation requirement
4. **Conversation design** — situational guidance mode, compare-traditions mode
5. **Web frontend** — simple, beautiful chat UI
6. **Waitlist / early users** — validate before building mobile
7. **Mobile app** — after web proves the product
8. **Optional fine-tuning** — RAFT on real query data
